## 1. 背景・目的

### 1.1 背景
中小規模の事業者では、顧客からの問い合わせを `info@` などの共有メールアドレスで複数人が対応している。
通常のメールクライアント（Gmail / Outlook）では以下の問題が起きる：

- 誰が対応中か分からず、**二重対応** または **対応漏れ** が発生する
- 過去のやり取りや対応履歴が個人の受信箱に分散する
- 返信品質が担当者によってばらつく
- 対応状況やKPI（応答時間など）が可視化されない

メールディーラー / Re:lation / yaritori / Zendesk 等がこの領域の既存製品。

### 1.2 目的
**「小規模チームが、迷わず・漏らさず・速く顧客対応できる」** ことを実現する。
さらに **AIが下書き・要約・分類を補助** することで、担当者の負荷を下げ、対応品質を底上げする。

### 1.3 プロダクトの差別化方針
- 既存大手より **軽量・シンプル・低価格・導入が速い**
- AIは**置き換えではなく補助**。送信は必ず人間が承認（信頼性・安心感を重視）
- 小さく始めて、中規模・マルチチャネルへ無理なく拡張できる設計

## 2. ターゲット・ペルソナ

### 2.0 サービスの3層構造（アクターモデル）
B2B SaaS の典型構造。混同しないこと。

| 層 | 説明 | 対応エンティティ | サーフェス |
|---|---|---|---|
| **事業管理者**（サービス提供者＝自社） | テナントの管理・サポート・課金 | （将来）`platform_admins` 等 | **別の super-admin 画面**（MVP外。当面はSupabaseダッシュボード/SQLで代用） |
| **テナント**（ユーザ組織＝顧客企業） | metasuke を使う会社。中に admin / member | `organizations` ＋ `memberships`(role) | **テナント用アプリ**（＝本MVPが作っているもの） |
| **顧客**（エンドカスタマー） | テナントにメールを送る/受け取る人。ログインしない | `contacts` | メール（受信箱経由） |

UX上の含意：
- テナントユーザーは通常 **1人＝1組織**。複数組織所属は例外（代理店等）なので、**組織が1つなら切替UIは出さない**。
- 組織作成は **サインアップ時のオンボーディング**で行う行為。常設ボタンにしない。
- 事業管理者の機能（全テナント横断の管理）は本アプリには載せず、別サーフェスで提供する。


### 2.1 直近ターゲット（優先）
個人〜小規模チーム（おおむね 1〜10名）。
EC事業者、SaaS / Webサービスの小規模サポート窓口、士業・小規模BtoBなど。

### 2.2 将来ターゲット（設計上は考慮）
中規模カスタマーサポート部門（10〜50名）。
→ 権限管理、レポート、承認フロー、SLA管理などが必要になる。
**今は作らないが、データモデル・権限設計は将来を塞がない形にしておく。**

### 2.3 主要ペルソナ
| ペルソナ | 役割 | 主なニーズ |
|---|---|---|
| オペレーター | 日々メール対応する担当者 | 二重対応を避けたい / 速く返信したい / 過去履歴をすぐ見たい |
| リーダー / 管理者 | チーム責任者 | 対応漏れを防ぎたい / 品質を揃えたい / 状況を把握したい |
| （将来）承認者 | 送信前チェック担当 | 新人の返信を確認してから送信したい |

## 3. スコープ

### 3.1 今回（MVP〜v1）対象
- **チャネル**: メール（メールSaaS経由の送受信）＋ 問い合わせフォーム受信
- **共有受信箱**: ステータス管理・担当者割り当て・二重対応防止
- **スレッド／顧客単位の履歴集約**
- **テンプレート・署名**
- **タグ・検索**
- **AI補助**: 返信下書き生成 / 要約 / 自動分類・タグ提案（すべて人間承認前提）
- **基本的なユーザー管理**（メンバー招待・ロール）

### 3.2 将来拡張（今回はやらない／設計で塞がない）
- マルチチャネル（**LINE** 優先、その後チャット・SNS等）
- 承認・ダブルチェックフロー
- SLA管理・高度なレポート / ダッシュボード
- FAQ／ナレッジベース（RAG）による回答候補提示
- AIによる半自動・自動返信
- 多言語・翻訳
- 外部連携（CRM、Slack通知など）

### 3.3 非対象（当面やらない）
- 電話・コールセンター機能（CTI）
- マーケティングメール配信（一斉配信）
- 本格的なヘルプデスク（チケット課金、資産管理など）

## 6. 機能要件（詳細） — ここから1機能ずつ深掘り

### 6.1 チケットのライフサイクル（F-04 ステータス管理）【ドラフト】

#### ステータス（MVPは4つに確定）
| ステータス | 意味 |
|---|---|
| `unassigned`（未対応） | 新規受信、まだ誰も担当していない。受信時の初期状態 |
| `open`（対応中） | 担当者が決まり対応している。または顧客から新着があり要対応 |
| `pending`（保留） | 顧客の返信待ち／社内確認待ち |
| `resolved`（対応済） | 対応完了。顧客返信があれば自動で `open` に戻る |

補足フラグ（ステータスとは別に内部で保持。将来の正式ステータス化を塞がない）:
- `replied`（最後にこちらが返信した）… 「返信済」絞り込みやレポートに使う内部フラグ
- `closed` / `spam` … 将来追加。MVPは `resolved` ＋ タグ/アーカイブで代用

#### 状態遷移ルール
- 新規メール受信 → `unassigned`
- 担当者をアサイン → `open`
- 返信送信 → `open` のまま（内部フラグ `replied=true` を立てる）
- 担当者が手動で `pending` / `resolved` に変更可
- 顧客から新着返信 → **`open` に戻し未読化＆担当者へ通知**（`resolved` でも自動再オープン）

### 6.2 共有受信箱・チケット一覧（F-03）【ドラフト】

#### ビュー（標準フィルタ）
- **未対応**（unassigned）

#### 一覧の1行に出す情報

#### ソート / 絞り込み
- ソート: 最終更新 / 受信日時 / 優先度（将来）
- 絞り込み: ステータス・担当者・タグ・Inbox・期間・未読

### 6.3 二重対応防止（F-06）【ドラフト】

二重対応はこのカテゴリの最重要価値。以下を組み合わせる。

1. **閲覧中インジケータ**: チケットを開いているメンバーのアバターをリアルタイム表示（"田中さんが閲覧中"）。
2. **返信中ロック**: 誰かが返信エディタを開くと「対応中（編集中）」を表示。他メンバーには警告を出す。
   - 方式は**ソフトロック**（強制排他ではなく警告）を既定とする。強制ロックは運用を硬くしがちなため。
3. **送信時の競合検知**: 送信直前に、自分が読み込んだ後に他者の新規メッセージ／ステータス変更がないかチェックし、あれば確認ダイアログ。
4. **アサインによる責任明確化**: 「自分にアサイン」で担当を可視化（運用上の二重対応防止の主軸）。

### 6.4 返信・送信フロー（F-08）＋ AI下書き（AI-01）【ドラフト】

```
チケットを開く
  └─ 過去履歴・顧客情報を表示
  └─ [AI下書き生成] ボタン → AIが文脈・履歴・テンプレを踏まえ下書きを生成
        └─ 下書きはエディタに挿入されるだけ（自動送信しない）★人間が必ず編集・承認
  └─ テンプレ挿入 / 署名付与 / 添付
  └─ [送信] → メールSaaS API経由で顧客ドメインから送信
  └─ 送信後ステータス自動更新（replied）＋履歴に記録（誰がいつ何を送ったか）
```

- AIの提案には「AI生成」のラベルを付け、編集された差分が分かるようにする。
- 送信前に**競合検知**（6.3-3）と**宛先確認**を挟む。

## 6.5 アーキテクチャ方針（確定事項）

### 提供形態: SaaS（マルチテナント）
- 複数の顧客企業（`Organization`）に提供する。**全データは `org_id` でテナント分離**する。
- 認証はメンバー（`User`）単位。組織ごとに招待・ロールを持つ。
- 実装初期は1組織でも、最初からマルチテナント前提のデータモデルにする。

### メール基盤: メールSaaS（Mailgun / Postmark / Resend 等）を利用
- 自前のメールサーバ（IMAP/SMTP常駐）は持たない。
- **受信（inbound）**: メールSaaSのInbound Webhookで受け取り、パースして `Message`→`Ticket` 化。
  - 各組織には受信用アドレス（例: `org内のInboxごとにエイリアス`）を割り当て、顧客はそこへ転送 or 独自ドメインのMXを向ける。
- **送信（outbound）**: メールSaaSの送信APIで送る。組織の独自ドメインを**ドメイン認証（SPF/DKIM/DMARC）**して、`From: support@顧客ドメイン` で送れるようにする。
- スレッド連結は `Message-ID` / `In-Reply-To` / `References` ヘッダで行う。

## 6.6 AI機能の詳細【ドラフト】

> 大方針：**AIは補助に徹し、最終的な送信は必ず人間が承認する**（[[人間承認]]の原則）。
> 利用基盤：**プロバイダ抽象化**。現状は **OpenAI** を採用（既存のOpenAI課金を利用。最安モデルを既定）。将来 Claude 等への差し替えが容易な構造。
> 費用モデル：**プラットフォーム共通キー＋組織別の月次上限／AI ON/OFF** を基本とし、**BYOK（テナントが自分のキーを設定・費用負担）** も機能として持つ（Phase 4b）。

### 6.6.1 モデル選定（現状）
| 用途 | モデル | 理由 |
|---|---|---|
| AI-01 返信下書き生成 | OpenAI 最安モデル（`gpt-4o-mini` 既定, env `OPENAI_MODEL`） | 下書きは最安モデルで十分なケースが多い。コスト最小化 |
| AI-02 スレッド要約（将来） | 同等の安価モデル | 要約は最高品質まで不要 |
| AI-03 分類・タグ/優先度提案（将来） | 同等の安価モデル | 高頻度・低レイテンシ。構造化出力で安定化 |

❓TODO: 品質が必要な組織向けに、上位モデルやプロバイダ（Claude等）を選べるようにする選択肢。

### 6.6.2 AI-01 返信下書き生成
**入力コンテキストの組み立て**（プロンプトに渡す材料）:
1. システムプロンプト（組織のトーン&マナー、禁止事項、署名ルール）… **固定・前方配置**
2. 関連ナレッジ／FAQ（将来のRAG。今は手動テンプレ）
3. 当該チケットの過去メッセージ（スレッド全体）
4. 顧客情報（Contact の属性、過去の対応傾向）
5. オペレーターの指示（任意。「丁寧に断る」「日程調整して」等の短い指示）

- **プロンプトキャッシュ**：1（システムプロンプト）と2（ナレッジ）は組織内でほぼ不変なので、プレフィックスをキャッシュしてコスト・レイテンシを削減。可変部（チケット本文・指示）は末尾に置く。
- **出力**：本文の下書き（複数案オプションも将来検討）。エディタに挿入されるのみで自動送信しない。
- **記録**：`ai_suggestions`（type=draft, model, accepted）。採否を蓄積し品質改善とコスト把握に使う。
- **生成方式**：現状は単発生成（非ストリーミング）。将来はストリーミング表示も検討。
- **上限・無効化**：組織単位の **AI ON/OFF（`organizations.ai_enabled`）** と **月次上限（`org_entitlements.monthly_draft_limit`）** を強制。未設定キーなら dry-run スタブ。

### 6.6.3 AI-02 スレッド要約
- 長い往復を3〜5行に要約。チケット詳細画面の冒頭に「AI要約」として表示（編集不可・参考情報）。
- 担当引き継ぎ時に効く。要約は `AISuggestion`(type=summary) に保存し再生成を避ける。

### 6.6.4 AI-03 自動分類・タグ/優先度提案
- 受信時に **構造化出力（JSON schema 強制）** で `{category, suggested_tags[], priority, sentiment}` を生成。
- あくまで**提案**。タグは自動付与せず「候補」を提示し、人間がワンクリックで確定。
- 高頻度なので OpenAI 最安モデル＋（将来）プロンプトキャッシュでコスト最小化。

### 6.6.5 AIプライバシー方針（要確定の重要事項）
顧客メール本文＝個人情報を外部LLMに送るため、以下を方針化する：
- **データ保持**：API のデータ保持・学習オプトアウト設定を前提に契約・設定する。
- **PIIマスキング**：必要に応じ送信前に電話番号・カード番号等をマスク（実装コストと精度のトレードオフ → 要検討）。
- **組織ごとのオプトアウト**：AI機能を組織単位でON/OFFできるようにする（AIに送りたくない顧客向け）。
- **監査**：AIへ送ったか／提案を採用したかを `AISuggestion`・`Event` に記録。

## 7. 非機能要件（後で詳細化）

- **マルチテナント分離**: 全クエリで `org_id` スコープを強制（行レベルの分離 or RLS）。テナント越境ゼロを保証。
- 性能 / 可用性 / セキュリティ（メール内容＝個人情報の保護、保存時/通信時の暗号化、操作監査ログ）
- **AIに関するプライバシー**: 顧客メール本文を外部LLMに送る是非・データ学習オプトアウト・PIIマスキングの方針 → 要決定。
- 到達率: 日本のキャリアメール（docomo/au/softbank）対応を考慮したプロバイダ選定。
- バウンス・苦情（complaint）処理、無限ループ（自動応答同士のループ）対策。

## 8. データモデル【ドラフト】

> 全テーブルに `org_id`（テナント分離キー）を持たせる。`id` はULID/UUIDを想定。

### 8.1 エンティティ関連（概念図）
```
Organization 1─┬─* User            （メンバー）
               ├─* Inbox           （共有受信箱: support@ など）
               ├─* Contact         （顧客）
               ├─* Tag / Template
               └─* Ticket 1─┬─* Message      （メール/フォームの各通）
                            ├─* Note         （内部メモ）
                            ├─* Event        （操作履歴/監査）
                            └─* AISuggestion （AI下書き・要約・分類提案）
Ticket *─1 Inbox / *─1 Contact / *─0..1 User(assignee)
```

### 8.2 主要テーブル（主なカラム）

**Organization**: `id`, `name`, `plan`, `created_at`

**User**: `id`, `org_id`, `email`, `name`, `role`(admin/member), `status`, `created_at`
- ❓TODO: 認証方式（自前 / Auth0等 / Clerk等）。SaaSなのでメール＋パスワード or OAuthログイン想定。

**Inbox**（共有受信箱）: `id`, `org_id`, `name`, `inbound_address`(受信用), `from_domain`, `signature`, `dkim_verified`
- 1組織が複数Inbox（例: support@ と sales@）を持てる。

**Contact**（顧客）: `id`, `org_id`, `email`, `name`, `company`, `meta`(任意属性), `created_at`
- 同一メールアドレスは1コンタクトに名寄せ。

**Ticket**（対応案件）: `id`, `org_id`, `inbox_id`, `contact_id`, `assignee_id`(nullable), `subject`, `status`(unassigned/open/pending/resolved), `replied`(bool), `is_read`, `priority`(将来), `last_message_at`, `created_at`, `closed_at`
- スレッド連結のため `thread_key`（Message-IDベース）を保持。

**Message**（各通）: `id`, `org_id`, `ticket_id`, `direction`(inbound/outbound), `channel`(email/form), `from`, `to`, `cc`, `subject`, `body_text`, `body_html`, `message_id`, `in_reply_to`, `references`, `attachments`, `sent_by`(outboundの送信者User), `created_at`
- メール固有ヘッダを保持し、返信時に正しくスレッドへ繋ぐ。

**Note**（内部メモ）: `id`, `org_id`, `ticket_id`, `author_id`, `body`, `mentions`[], `created_at`（顧客には見えない）

**Tag**: `id`, `org_id`, `name`, `color` ／ 中間表 `ticket_tags`

**Template**: `id`, `org_id`, `name`, `subject`, `body`, `inbox_id`(nullable), `created_at`

**AISuggestion**: `id`, `org_id`, `ticket_id`, `type`(draft/summary/classification), `input_hash`, `output`, `model`, `accepted`(bool), `created_at`
- 提案の採否を記録し、品質改善・コスト把握に使う。

**Event**（操作履歴・監査ログ）: `id`, `org_id`, `ticket_id`, `actor_id`, `type`(assigned/status_changed/sent/note_added/...), `payload`, `created_at`
- 「誰がいつ何をしたか」を全て記録。二重対応検知・レポート・監査の基盤。
