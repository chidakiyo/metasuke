# metasuke — 事業管理者（プラットフォーム運営）サーフェス設計（ドラフト v0.1）

> 3層アクターモデルの最上位「事業管理者（サービス提供者＝自社）」が使う運営コンソールの設計。
> テナント用アプリ（`apps/web`）とは**別物**として安全に作る。
> ステータス: 🟡 設計ドラフト / 最終更新: 2026-06-24

---

## 1. 位置づけ・原則

- 対象ユーザー：**自社の運営/サポート/課金担当**（テナントの顧客ではない）。
- テナントアプリとは分離した**別サーフェス**（別ルート `/admin` or 別アプリ、できれば別サブドメイン）。
- 設計原則（他社事例＋セキュリティのベストプラクティスより）：
  1. **小さく始める**。日々繰り返す運用（テナント調査・パスワード再設定・利用状況確認）から。高度な分析や複雑ワークフローは後回し。
  2. **`service_role`（全権キー）は絶対にブラウザに出さない**。横断データ取得は必ず Edge Function 側（サーバ）で実行し、呼び出し元が運営者か検証してから返す。
  3. **テナント識別子は常に認証セッション由来**。リクエストボディの `org_id` を信用しない（越境防止／OWASP）。
  4. **全運営操作を不変の監査ログへ**（誰が・いつ・どのテナント/ユーザーに・何を・なぜ）。特に代理ログイン。
  5. **最小権限**。運営者にもロール（superadmin / support / billing / readonly）。

## 2. 機能セット（他社の定番を踏まえて）

> 出典の定番：アカウント検索＋プロファイル（サブスク/利用/課金を1画面）、基本操作（プラン変更・クレジット付与・トライアル延長・アカウント停止）、アカウント単位のフィーチャーフラグ、利用状況/上限の可視化、サポート文脈の統合、監査ログ。これらを metasuke 向けに具体化する。

| # | 機能 | 内容 | 優先 |
|---|---|---|---|
| PA-01 | **テナント一覧・検索** | 全テナント（organizations）を名前/ドメイン/IDで検索。プラン・状態・メンバー数・最終アクティブ | ◎ |
| PA-02 | **テナント詳細** | 1画面でサブスク/利用状況/メンバー/受信箱/直近アクティビティ | ◎ |
| PA-03 | **利用状況メトリクス** | チケット数・送受信メール数・AI利用量(トークン/件数)・ストレージ。課金とサポートの根拠 | ◎ |
| PA-04 | **監査ログ（運営操作）** | 運営者の全操作を不変記録・検索 | ◎ |
| PA-05 | **運営者管理・ロール** | platform_admins、ロール(superadmin/support/billing/readonly)、2FA必須 | ◎ |

## 3. セキュリティ・アーキテクチャ

```
[運営者ブラウザ: admin SPA]   ← 2FA必須・IP許可リスト（任意）
        │ 運営者JWT
        ▼
[admin Edge Functions (Hono)]   ← ここだけが service_role を使う
        │ 1) 呼び出し元が platform_admins か検証
        │ 2) 操作を platform_audit_log に必ず記録
        │ 3) 横断クエリ（全org）を service_role で実行
        ▼
[Supabase: Postgres / Auth / Storage]
```

- **横断アクセスの実現**：テナントアプリのRLSは越境ゼロ。運営は service_role でRLSを迂回するが、**サーバ側関数の中だけ**。ブラウザには結果（最小限）だけ返す。
- **運営者認証**：Supabase Auth を流用しつつ、`platform_admins` に登録された人だけ admin サーフェスに入れる。**2FA必須**、可能なら**別ドメイン/別アプリ**でCookie/セッションを分離。
- **代理ログイン（impersonation）の安全策**（最重要・要注意機能）：
  - **理由の入力を必須**にしてから開始。
  - **短い時間制限**（例: 30分）と明示的終了。
  - 代理中は画面に**常時バナー**（"サポートとして閲覧中"）。
  - **既定は読み取り専用**。書き込みを許す場合はさらに上位権限＋追加記録。
  - **専用の監査記録**（対象org/user・理由・開始/終了・実行した操作）。テナント側へ通知するかは方針決定。
  - 実装はテナントのパスワードを使わず、**サーバが短命の代理トークンを発行**して該当orgスコープで動かす。
- **監査ログは追記専用・改ざん不可**（UPDATE/DELETE禁止）。tenant/org文脈を必ず含める。
- **最小権限**：readonly運営者は閲覧のみ、support は代理/サポート操作、billing は課金、superadmin のみ破壊的操作。
- **コンプライアンス**：個人情報（メール本文）アクセスは記録し、GDPR/個人情報保護の観点で代理閲覧・エクスポート・削除を統制。

## 4. データモデル追加（案）

```
platform_admins        (user_id PK → auth.users, role, status, created_at)
                       role: superadmin | support | billing | readonly
platform_audit_log     (id, actor_id, action, target_org_id, target_user_id,
                        reason, payload jsonb, ip, user_agent, created_at)  -- 追記専用
impersonation_sessions (id, admin_id, target_org_id, target_user_id, reason,
                        started_at, expires_at, ended_at, readonly bool)

-- organizations に運営用カラムを追加
organizations += plan, status(active/suspended/closed), trial_ends_at,
                 suspended_reason, created_by

-- エンタイトルメント（テナント単位の機能/上限）。ai_enabled は既存。
org_entitlements       (org_id PK, ai_enabled, monthly_ai_token_limit,
                        seat_limit, inbox_limit, feature_flags jsonb)
```

- これらは **RLSで一般テナントから不可視**にする（platform_admins 専用 or service_role のみ）。
- 利用状況メトリクスは既存テーブル（tickets/messages/ai_suggestions）の集計か、別途集計テーブル/マテビューで提供。

## 5. 段階導入プラン

- **Phase A（最小・土台）**：別ルート/別アプリの admin サーフェス雛形 ＋ `platform_admins` ＋ admin Edge Function（service_role・運営者検証・監査記録）＋ **PA-01/02/03/04/05**（テナント一覧・詳細・利用状況・監査・ロール）。
- **Phase B**：**PA-06 代理ログイン**（厳重な監査・時間制限・読み取り専用）。
- **Phase C**：**PA-07/08/09/10**（テナント操作・フィーチャーフラグ・Stripe課金・お知らせ）。
- **Phase D**：**PA-11/12/13**（不正検知・ヘルス・サポート文脈）。

> ※ これはテナント向けMVP（Phase 0〜4）とは**別トラック**。テナントMVP完成後、または運営負荷が上がってきた段階で着手するのが定石（「小さく始め、実運用の必要が出たら育てる」）。

## 6. 設計判断メモ
- **当面は未実装でも問題ない**（少数テナントなら Supabase ダッシュボード/SQL で代用可能）。テナント数が増え、サポート/課金/濫用対応が手作業で回らなくなったら Phase A から。
- **作るなら最初から分離**（別サーフェス・service_roleはサーバ限定・監査必須）。テナントアプリに混ぜると事故時の被害が全テナントに及ぶ。
- 代理ログインは価値が高いが**最も危険**。導入時は監査・権限・通知を必ずセットで。
