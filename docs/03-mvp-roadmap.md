## 1. MVPの定義（ゴール）

> **「小規模チームが、顧客メールを共有受信箱で受け、漏らさず・二重対応せず、AI下書き付きで返信できる」**

これを満たす**最小の縦切り（受信→表示→対応→返信）**をマルチテナントSaaSとして成立させる。
差別化点である **AI下書き（AI-01）** までを MVP に含める。

### MVP完了の受け入れ基準（Acceptance）
1. 組織を作りメンバーを招待でき、データは `org_id` で完全分離（他組織のデータが一切見えない）。
2. 顧客が組織の受信アドレスへ送ったメールが、共有受信箱に**チケット**として現れる。
3. メンバーがステータス（未対応/対応中/保留/対応済）を変更でき、担当者をアサインできる。
4. 同じチケットを他メンバーが開くと**閲覧中/対応中が見える**（二重対応防止）。
5. スレッド（同一顧客の往復）が1画面に集約表示される。
6. **AIが下書きを生成**し、メンバーが編集・承認して**顧客ドメインから送信**できる（自動送信はしない）。
7. デプロイは `supabase` CLI で完結する。

## 3. ロードマップ（フェーズ）

各フェーズは「動いて確認できる単位」。上から順に依存。

### Phase 0 — 基盤（土台が立つ）
- モノレポ scaffold（`apps/web`, `supabase/functions`, `packages/shared`）
- Supabaseプロジェクト作成、Auth有効化
- `Organization` / `User(membership)` / ロール、RLSポリシーの雛形
- ログイン → 組織作成 → メンバー招待 のフロー
- ✅ 確認: 2組織を作り、互いのデータが見えないこと（RLS）

### Phase 1 — 受信して見える（read path）
- データモデル：`Contact` / `Ticket` / `Message` とRLS
- メールSaaS設定（受信アドレス、DKIM）、inbound webhook（保存＋即200）
- スレッド連結（Message-ID/In-Reply-To）
- 共有受信箱の一覧 ＋ チケット詳細（スレッド集約表示）
- ✅ 確認: 受信アドレスへ送ったメールが一覧とスレッドに現れる

### Phase 2 — 対応できる（write path）
- ステータス変更（4種）・担当者アサイン
- 返信エディタ → メールSaaS送信API（顧客ドメイン送信、In-Reply-To付与）
- 送信後のステータス自動更新、`Event` 記録、バウンス受信の最低限処理
- ✅ 確認: アサイン→返信送信→顧客に届く、履歴が残る

### Phase 3 — 二重対応を防ぐ
- Supabase Realtime で在席（閲覧中/編集中）表示
- ソフトロック（警告）と、送信直前の競合検知
- ✅ 確認: 2人で同じチケットを開くと相手の在席が見える／競合時に警告

### Phase 4 — AIが助ける（差別化）
- `ai_suggestions` テーブル、OpenAI呼び出し（プロバイダ抽象化・dry-runフォールバック）
- 「AI下書き生成」ボタン → エディタに挿入（自動送信なし）→ 編集して送信
- 採否を記録、組織単位のAI ON/OFF
- ✅ 確認: 下書きが文脈を踏まえて生成され、人が承認して送れる
- **← ここまでで MVP 完了**

### Phase 5 以降 — v1の肉付け（任意・優先度順）
- F-02 フォーム受信 → F-09 テンプレ/署名 → F-10 タグ → F-11 検索 → F-12 内部メモ → F-14 通知
- AI-02 要約 → AI-03 自動分類
- 複数受信箱、レポート、承認フロー …
